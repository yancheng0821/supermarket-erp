# 登录入口重构与首页 Tab 体验升级设计

## 背景

当前前端存在两个直接影响产品感的问题：

1. 默认登录页只有一个居中窄卡片，信息层级很薄，看起来更像组件库示例页，不像生产环境的 ERP 产品入口。
2. 登录表单内部通过基础 `Tabs` 在“租户管理员 / 平台管理员”之间切换，交互生硬，角色选择缺少明确分层。
3. 首页 dashboard 顶部的 tab 也沿用了同一套基础样式，切换状态、反馈和内容承接都不自然。

用户要求参考 Mobbin 这类成熟产品的交互基调，将登录入口改造成“先选角色，再进入表单”，同时把首页 tab 的切换提升到更接近生产产品的质量。

## 目标

- 将默认登录页重构为双入口角色选择页。
- 将登录流程拆分为“主入口页 -> 对应角色登录页”。
- 移除登录表单内部的角色切换 tab。
- 将 dashboard 顶部 tab 升级为更顺滑、更有状态感的 segmented control 风格切换。
- 维持现有登录 API、权限体系和路由守卫逻辑，不修改后端协议。

## 非目标

- 不改动登录接口字段与返回结构。
- 不新增第三种登录角色。
- 不重做整个后台首页信息架构。
- 不在本轮引入动画库。
- 不处理 dashboard 真实数据接入，仅优化现有静态展示层和交互层。

## 设计方向

采用“C：登录更有品牌感，后台回到企业控制台”的折中路线，并落地为如下视觉策略：

- 登录主入口页：暖白 + 浅灰绿的氛围底色，双入口卡片建立角色分层，增加少量品牌说明与信任信息。
- 角色登录页：收敛为更稳重的企业风格，保留品牌识别，但主体是专业的双栏登录布局。
- 后台 dashboard：弱化营销气质，使用更克制的企业感 segmented control 与内容区过渡。

## 信息结构

### 1. 主登录页

页面不再直接展示用户名密码表单，而是展示两个明确入口卡片：

- 租户管理员
- 平台管理员

每个入口卡片包含：

- 角色标题
- 一句职责说明
- 主按钮
- 与角色关联的业务关键词

页面上方补充品牌标题与简短产品说明，用于建立“这是一个正式的企业系统入口”的感知。

### 2. 角色登录页

用户点击某个入口后，再进入对应角色的登录页。

角色登录页包含：

- 返回主入口页的返回动作
- 当前角色名称与描述
- 对应的登录表单
- 表单右侧或下方的品牌/价值摘要

交互上，不再允许通过 tab 直接在角色之间横跳；切换角色只能回到主入口页重新选择。

### 3. Dashboard 首页

dashboard 顶部主切换只保留已有内容的两个视图：

- 总览
- 分析

未完成内容如“报表”“通知”不再以 disabled tab 的形式暴露在主切换区，避免占位控件破坏完成度。

## 交互设计

### 登录入口页

- 卡片整体可点击，按钮作为强化动作，不制造“双热点冲突”。
- hover / focus 时卡片边框、阴影与背景有轻微变化，传达可进入性。
- 两张卡片的视觉主次不同，但都保持企业产品的克制。

### 角色登录页

- 从入口页进入登录页时，保留角色连续性：标题、说明、按钮文案与返回路径都应该明确。
- 租户登录保留 `tenantCode` 字段。
- 平台登录不展示 `tenantCode` 字段。
- 表单错误提示、加载状态与成功提示沿用现有能力，但视觉层级要更清晰。

### Dashboard segmented control

- 顶部切换条采用容器底板 + 激活滑块 + 文本前景变化的结构。
- 切换反馈应在 150-220ms 范围内，避免生硬闪烁。
- 内容区切换采用轻量淡入/位移，不使用夸张动画。
- 切换器需要在移动端可横向容纳，并保持可点击区域足够大。

## 技术设计

### 路由策略

保留现有 `/sign-in` 路径作为默认登录入口。

新增两个角色登录子路由，建议如下：

- `/sign-in/tenant`
- `/sign-in/platform`

理由：

- 语义清晰，便于分享和回跳。
- 保持与现有认证路由结构一致。
- 方便主入口页只负责选择，角色页只负责表单。

### 组件拆分

建议按责任拆分：

- `features/auth/sign-in/index.tsx`
  - 主登录入口页
- `features/auth/sign-in/components/login-entry-card.tsx`
  - 角色入口卡片
- `features/auth/sign-in/role-sign-in.tsx`
  - 通用角色登录页外壳
- `features/auth/sign-in/components/user-auth-form.tsx`
  - 纯表单组件，改为通过 `mode` prop 驱动
- `features/auth/auth-layout.tsx`
  - 升级为可支持入口页与角色登录页的 auth shell
- `components/ui/segmented-control.tsx` 或在现有 `tabs.tsx` 上扩展变体
  - dashboard 使用的新切换视觉样式

### 状态策略

登录表单不再内部维护角色选择状态，而是由路由或父组件传入：

- `mode='tenant'`
- `mode='platform'`

这样可以：

- 去掉表单内部 tab 逻辑
- 让页面结构更清晰
- 方便测试单个角色页渲染

### 样式策略

不引入新 UI 库。

延续现有 Tailwind + shadcn 体系，但为登录页与 dashboard 增加更明确的语义视觉：

- 登录入口页：定制背景层、角色卡片、信息区
- 角色登录页：定制双栏结构与卡片层级
- dashboard：定制 segmented control 风格

## 数据与接口影响

本轮不修改：

- `platformLogin`
- `tenantLogin`
- `bootstrapSession`
- `auth-store`
- route guard 的权限判定逻辑

唯一变化是前端进入这些 API 的路径从“表单内切换 mode”变为“由外层页面决定 mode”。

## 测试策略

至少补充以下验证：

1. 主登录页渲染两个角色入口。
2. 点击租户入口进入租户登录页。
3. 点击平台入口进入平台登录页。
4. 租户登录页展示 `tenantCode` 字段，平台登录页不展示。
5. dashboard 只展示“总览 / 分析”两个主切换项。
6. 切换内容区域时对应面板正确渲染。

## 风险与应对

### 风险 1：路由改动影响现有未登录跳转

应对：

- 保持 `/sign-in` 作为默认入口不变。
- 只有新增角色子路由，不替换现有守卫逻辑。

### 风险 2：视觉改动过重，和现有后台风格割裂

应对：

- 将品牌感集中在登录入口页。
- 角色登录页与后台首页统一回到更克制的企业风格。

### 风险 3：dashboard tab 重构影响其他页面通用 tabs

应对：

- 优先新增 dashboard 专用变体或轻量封装。
- 不全局替换现有 `Tabs` 默认样式，避免波及其他页面。

## 验收标准

- `/sign-in` 成为双入口角色选择页。
- 角色表单不再通过 tab 切换。
- 登录页视觉层级明显提升，具备生产产品首页入口感。
- dashboard 顶部切换不再展示 disabled tab，占位项移除。
- dashboard 切换的状态反馈与内容承接更自然。
- 前端测试、lint、build 保持通过。
